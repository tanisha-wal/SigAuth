1. Project Overview
This document is the single source of truth for the Internal Identity Provider (IdP) intern project. The platform is built around the OIDC ID Token as the exclusive mechanism for user authentication. Access Tokens and Refresh Tokens, while described in this document, are designated as optional extensions. Organization management is likewise optional and earns bonus marks.

1.1 Objectives
	•	Authenticate users securely with email + password using the OIDC Authorization Code flow.
	•	Issue a signed OIDC ID Token as the single proof of identity consumed by client applications.
	•	Delegate login to a centralised IdP — no per-app credential storage.
	•	Implement full RBAC using Groups as the assignment vehicle; embed roles and permissions inside the ID Token.
	•	Register and manage applications (OAuth clients) that redirect users to this IdP for login.
	•	Provide a React admin console for managing users, groups, roles, and applications.
	•	[Optional] Support multi-tenant Organizations with isolated resources.
	•	[Optional] Issue Access Tokens and Refresh Tokens for API-level authorisation.

1.2 Problem Statement
Enterprise teams using separate per-app login systems face credential sprawl, inconsistent access policies, and audit gaps. This project builds a simplified internal identity provider — inspired by Okta and Auth0 — that centralises authentication in one service. Client applications receive an OIDC ID Token after login; they parse the token's claims to determine who the user is and what they are allowed to do, without any further round-trips to the IdP.

2.3 ID Token as Authentication Proof
When a client application receives an ID Token after the OIDC Authorization Code flow, it MUST verify the token before trusting any claim inside it. The verification steps are:
	•	Fetch the IdP's public key from GET /.well-known/jwks.json using the kid in the token header.
	•	Verify the RS256 signature using the matching public key.
	•	Confirm the iss (issuer) matches the expected IdP URL.
	•	Confirm the aud (audience) matches the application's own client_id.
	•	Confirm the exp (expiry) has not passed.
	•	If the authorization request included a nonce, verify the nonce claim matches.
Once verified, the application reads sub (user identity), email, roles, and permissions directly from the ID Token — no further IdP call is required.

2.4 Service Boundaries
	•	Auth API — Public-facing endpoints consumed by browsers and client apps: /authorize, /token, /userinfo, JWKS, OIDC discovery.
	•	Management API — Admin-protected endpoints for the React Console: user CRUD, group management, application registry.


3. Database Schema
All tables use UUID primary keys and include created_at / updated_at timestamps. Soft-delete (deleted_at) is preferred over hard deletes for audit integrity. The org_id column is included in all tables to support optional multi-tenancy; when the Organization module is not implemented, all rows can share a single default org UUID.

3.1 Organizations (supports optional multi-tenancy)
CREATE TABLE organizations (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name          TEXT NOT NULL UNIQUE,
  slug          TEXT NOT NULL UNIQUE,
  display_name  TEXT,
  status        TEXT NOT NULL DEFAULT 'active',
  settings      JSONB DEFAULT '{}',
  created_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
  deleted_at    TIMESTAMPTZ
);
-- When Org module is not implemented: INSERT a single default org and use its id everywhere.

3.2 Users

  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),

3.7 ID Tokens (Required) + Optional Access / Refresh Tokens

6. User Management & Group-Based RBAC
Role-Based Access Control is implemented through Groups. Users are never assigned roles directly — users join Groups, and Groups hold Roles. This indirection makes bulk permission changes simple and produces clean audit trails.

Design Principle:  Users → Groups → Roles → Permissions. Roles and permissions are resolved at ID Token issuance and embedded as claims. Client applications read claims from the token — no extra API call needed.

6.5 Effective Permissions Resolution (at ID Token Issuance)
	•	Fetch all groups the user belongs to.
	•	Collect all roles assigned to those groups (de-duplicated).
	•	Expand each role into its permission strings.
	•	Embed the union set into the ID Token as roles and permissions claims.

-- Run this query when issuing the ID Token
SELECT DISTINCT unnest(r.permissions) AS permission
FROM   group_members gm
JOIN   group_roles   gr ON gr.group_id = gm.group_id
JOIN   roles         r  ON r.id = gr.role_id
WHERE  gm.user_id = :user_id
  AND  r.org_id   = :org_id;

7. Authentication & Authorization Flows
★  CORE AUTH PRINCIPLE — Authentication is performed exclusively using the OIDC ID Token. Access Tokens and Refresh Tokens are optional extensions.

7.1 Core Flow — OIDC Authorization Code + PKCE
This is the only required authentication flow. The result is an OIDC ID Token which the client application uses as proof of the user's identity.

	•	Client generates a code_verifier (random 43–128 char string) and derives code_challenge = BASE64URL(SHA256(code_verifier)).
	•	Client generates a nonce (random string) and stores it locally for replay protection.
	•	Client redirects user browser to:

GET /api/v1/authorize
  ?response_type=code
  &client_id=<client-id>
  &redirect_uri=https://app.internal/callback
  &scope=openid profile email
  &state=<random-csrf-state>
  &nonce=<random-nonce>
  &code_challenge=<s256-hash>
  &code_challenge_method=S256

	•	IdP displays the login page. User enters email and password.
	•	IdP validates credentials, resolves RBAC claims via group query, stores authorization code (with nonce + code_challenge), then redirects browser to redirect_uri?code=...&state=...
	•	Client verifies state matches. Client POSTs to /token:

POST /api/v1/token
Content-Type: application/x-www-form-urlencoded

grant_type=authorization_code
&code=<auth-code>
&redirect_uri=https://app.internal/callback
&client_id=<client-id>
&code_verifier=<original-verifier>

	•	IdP verifies code_challenge matches SHA256(code_verifier). IdP issues the ID Token. If the optional Access Token module is enabled, an access_token is also returned.
	•	Client verifies the ID Token (see Section 2.3). Client reads sub, email, roles, permissions from the token claims.
	•	Client stores the verified ID Token (e.g., in memory or HttpOnly cookie) and uses its claims for all in-app authorisation decisions.

7.2 ID Token — Full Claim Set
The ID Token is a signed JWT containing standard OIDC claims plus custom RBAC claims.

// ID Token Header
{
  "alg": "RS256",
  "kid": "key-id-1",
  "typ": "JWT"
}

// ID Token Payload
{
  // Standard OIDC claims
  "iss":            "https://auth.internal",           // Issuer
  "sub":            "user-uuid-alice",                 // Subject (user ID)
  "aud":            "client-id-hr-portal",             // Audience (must match client_id)
  "iat":            1700000000,                        // Issued at
  "exp":            1700003600,                        // Expiry
  "nonce":          "abc123xyz",                       // Replay protection
  "jti":            "unique-token-id",                 // JWT ID for revocation

  // Identity claims (from openid, profile, email scopes)
  "email":          "alice@acme.com",
  "email_verified": true,
  "name":           "Alice Smith",
  "given_name":     "Alice",
  "family_name":    "Smith",

  // Custom RBAC claims (resolved from Groups)
  "org_id":         "org-uuid-acme",                   // Present if Org module enabled
  "roles":          ["app:manager", "viewer"],
  "permissions":    ["app:create", "app:update", "user:read", "group:read"]
}

7.3 Token Endpoint Response
The /token endpoint always returns an ID Token. Access and Refresh Tokens are only included when those optional modules are implemented and the application has refresh_token_enabled = true.

// Core response (ID Token only)
{
  "id_token":    "<signed-jwt>",   // Always present
  "token_type":  "Bearer",
  "expires_in":  3600
}

// Extended response (optional Access + Refresh tokens enabled)
{
  "id_token":      "<signed-jwt>",
  "access_token":  "<signed-jwt>",  // Optional
  "refresh_token": "<opaque>",       // Optional
  "token_type":    "Bearer",
  "expires_in":    3600
}

7.4 OIDC UserInfo Endpoint
The /userinfo endpoint accepts the ID Token as the Bearer credential and returns the user's current claims. This is useful when the client needs fresher claims than those in a cached ID Token.
GET /api/v1/userinfo
Authorization: Bearer <id-token>

Response 200:
{

  "email":          "alice@acme.com",
  "email_verified": true,
  "name":           "Alice Smith",
  "roles":          ["app:manager"],

}

7.5 [Optional] Access Token & API Authorisation
⚑  OPTIONAL MODULE — Access Token  — Implementation of this section is not required for the core project. It earns bonus marks.

When implemented, resource servers (APIs) accept the Access Token in the Authorization: Bearer header to authorise API calls. The Access Token is a separate JWT from the ID Token and carries the same RBAC claims but with an audience set to the resource server, not the client app.

Key Distinction:  The ID Token proves WHO the user is (authentication). The Access Token proves what the user may do at a specific API (authorisation). Do not use ID Tokens as access credentials for APIs.

7.6 [Optional] Refresh Token
⚑  OPTIONAL MODULE — Refresh Token  — Implementation of this section is not required for the core project. It earns bonus marks.

Refresh tokens allow the client to obtain new ID Tokens (and optionally Access Tokens) without prompting the user to log in again. When implemented:
	•	Refresh token rotation must be applied — each use issues a new refresh token and invalidates the old one.
	•	Reuse detection: if an already-used refresh token is presented, revoke the entire token family immediately.
	•	Refresh tokens are opaque strings — never JWTs — and must be stored server-side.

POST /api/v1/token
Content-Type: application/x-www-form-urlencoded

grant_type=refresh_token
&refresh_token=<opaque-refresh-token>
&client_id=<client-id>

9. Security Requirements
9.1 Password Handling
	•	Use bcrypt with cost factor >= 12.
	•	Never store or log plaintext passwords.
	•	Enforce: 8+ chars, mixed case, at least one digit or special character.
	•	Account lockout after 5 consecutive failures (exponential back-off or 15-min lock).

9.2 ID Token Security
	•	Sign ID Tokens with RS256 (asymmetric). Never use HS256 — all services would share the signing secret.
	•	Set exp to a short lifetime (15–60 minutes). Clients must re-authenticate when the ID Token expires.
	•	Always include and verify the nonce claim to prevent replay attacks.
	•	Store issued jti values; check against a revocation set on /userinfo and /introspect calls.
	•	The aud claim MUST be the requesting application's client_id — reject tokens intended for other apps.

9.3 Optional Token Security (Access / Refresh)
⚑  OPTIONAL MODULE — Access Token & Refresh Token Security  — Implementation of this section is not required for the core project. It earns bonus marks.

	•	Access Tokens should have a separate aud set to the resource server URL, not the client app.
	•	Refresh tokens must be rotated on every use. Store only the hash, never the raw value.
	•	Implement refresh token family revocation: if a reused refresh token is detected, revoke all tokens in the family.

9.4 Transport & Infrastructure
	•	Enforce HTTPS everywhere; reject HTTP at the load balancer.
	•	CORS policy: allow only origins matching registered redirect_uri values.
	•	Rate limiting: 10 login attempts per IP per minute; 100 /token requests per client per minute.
	•	Validate redirect_uri as exact-match against the whitelist — no partial matches, no open redirects.
	•	Set SameSite=Strict; Secure; HttpOnly on any session cookies.

9.5 PKCE
Mandatory:  SPA and native app types MUST include code_challenge (S256). The IdP MUST reject authorization requests from these types that omit code_challenge.

9.6 MFA (Optional — Bonus)
	•	Implement TOTP (RFC 6238) using pyotp or a similar library.
	•	On MFA setup, display backup codes (store as bcrypt hashes).
	•	Enforce org-wide via Organization settings enforce_mfa flag (requires Org module).

11. Additional Implementations
11.1 Email Verification
	•	On user creation, send a verification email with a signed HMAC-SHA256 link valid for 24 hours.
	•	GET /api/v1/verify-email?token=<signed-token> sets email_verified = true.
	•	When org setting require_email_verification is true, block login for unverified accounts.

11.2 Password Reset Flow
	•	POST /api/v1/password-reset/request  { email }
	•	IdP generates single-use UUID token (DB row with 1-hour expiry) and emails the reset link.
	•	POST /api/v1/password-reset/confirm  { token, new_password }
	•	IdP validates token, hashes new password, saves it, invalidates all existing sessions.

11.3 Session Management via ID Token
Sessions are tied to the ID Token's jti. Since ID Tokens are short-lived, sessions naturally expire. For explicit revocation:
	•	Store session metadata (jti, user_id, client_id, ip_address) in Redis with TTL = token expiry.
	•	GET /api/v1/me/sessions — list active sessions for the current user.
	•	DELETE /api/v1/me/sessions/:jti — revoke a specific session by its token's jti.
	•	POST /api/v1/organizations/:org_id/users/:id/revoke-sessions — admin revokes all sessions for a user.

11.4 JWKS Endpoint & Key Rotation
	•	Publish public RS256 keys at GET /.well-known/jwks.json. This allows any client to verify ID Tokens locally without calling the IdP.
	•	Each key has a kid; the same kid appears in every ID Token's header.
	•	Key rotation: publish both old and new keys for a 24-hour overlap window, then retire the old key.

11.5 OIDC Discovery Document
Publish /.well-known/openid-configuration so clients can auto-configure. Required fields: issuer, authorization_endpoint, token_endpoint, userinfo_endpoint, jwks_uri, scopes_supported, id_token_signing_alg_values_supported (must include RS256).

14. Glossary
Term
Definition
IdP
Identity Provider — a service that authenticates users and issues identity tokens.
OIDC
OpenID Connect — an identity layer on top of OAuth 2.0; defines the ID Token.
ID Token
A signed JWT issued by the IdP after authentication. Contains the user's identity and RBAC claims. The primary (and only required) token in this platform.
OAuth 2.0
An authorization framework (RFC 6749) enabling delegated access.
JWT
JSON Web Token (RFC 7519) — a compact, signed token encoding claims.
PKCE
Proof Key for Code Exchange (RFC 7636) — required for public clients.
RBAC
Role-Based Access Control — permissions assigned via Groups, embedded in the ID Token.
nonce
A random value included in the authorization request and mirrored in the ID Token to prevent replay attacks.
iss
JWT issuer claim — the IdP URL. Clients verify this matches the expected value.
sub
JWT subject claim — the user's unique identifier.
aud
JWT audience claim — must equal the application's client_id.
jti
JWT ID — unique per-token identifier used for revocation tracking.
JWKS
JSON Web Key Set — the IdP's public keys, used by clients to verify ID Token signatures.
Access Token
[Optional] An OAuth 2.0 JWT authorising calls to a specific API/resource server.
Refresh Token
[Optional] Opaque credential used to renew ID/Access tokens without re-login.
Organization
[Optional] Tenant boundary; all resources are scoped to an org.
Group
A named collection of users; the bridge between users and roles.
Role
A named set of permissions assigned to groups, then embedded in tokens.
client_id
Public identifier for a registered application.
client_secret
Confidential credential for server-side (web/m2m) applications.
